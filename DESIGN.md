# 莞中人飞跃手册视觉契约

> 这份文件约束飞跃手册主站、投稿模板和公开转载物料的共同气质。它借鉴 Kami 的纸感与克制，也借鉴 open-design 的 `DESIGN.md` 思路，但不照搬任何现成系统。

## 设计目标

飞跃手册应该像一份认真编辑过的校友手册：温暖、清楚、可信、愿意被保存。它不是营销落地页，也不是后台管理系统。

读者打开页面时，第一感受应该是：

- 这是莞中校友认真整理出来的东西
- 内容有层次，不是公众号碎片合集
- 可以长期查、反复看、转发给同学

## 视觉关键词

- 克制
- 纸感
- 编辑部
- 校友手册
- 中文长文友好
- 低噪音

## Kami / Open Design 落地理解

这套视觉不是“给 MkDocs 加一点颜色”，而是把页面当成一张认真排版的纸。

从 Kami 借鉴：

- 页面首先是一张纸，不是 SaaS 控制台。
- 主色只承担少量强调，不能到处抢戏。
- 字体层级和留白比卡片装饰更重要。
- 阴影、渐变、彩色提示块都要克制。
- 状态说明更适合做成细小的版面信息，而不是 quote 式大框。

从 open-design 借鉴：

- `DESIGN.md` 必须是可执行约束，不只是审美愿望。
- 组件命名、颜色、间距、禁忌要能被人和 agent 反复引用。
- 页面先定义结构角色，再决定视觉样式。
- MVP 也要有产品气质：空状态可以简单，但不能像临时占位。

从 Discourse 借鉴：

- 这是一个社区产品，不只是静态文档。
- 目录入口应像“话题列表”：类别、状态、可点击区域和讨论入口要清楚。
- 评论区是社区参与入口，视觉上应被纳入页面系统，而不是外挂脚本后的说明文字。
- 移动端不能隐藏关键功能；入口要有足够触摸面积。
- 社区状态要轻、稳定、可扫描，不使用花哨卡片堆叠。
- 不使用左侧粗色条、侧边竖线或“边框高亮”作为主要视觉语言；这会显得像 AI 常见模板，也会压迫移动端阅读。

## 色彩

第一版采用暖纸色背景和深墨蓝重点色，避免纯白、纯黑和高饱和渐变。

```text
paper-bg:       #f6f1e8
paper-surface:  #fffaf0
ink-main:       #202124
ink-muted:      #5f6368
ink-blue:       #1b365d
line-warm:      #ded4c3
soft-green:     #5f7f72
soft-red:       #9b4a3f
```

使用规则：

- 背景用 `paper-bg`，正文容器用 `paper-surface` 或透明。
- 链接、导航高亮、重点按钮用 `ink-blue`。
- `soft-green` 用于成功、可行动提示、贡献入口。
- `soft-red` 只用于风险、边界、隐私提醒。
- 不使用紫蓝渐变、玻璃拟态、荧光边框。

## 字体

网页第一版优先稳定和可读，不追求稀有字体。

```text
heading: "Noto Serif SC", "Source Han Serif SC", serif
body:    "Noto Sans SC", "Source Han Sans SC", system-ui, sans-serif
code:    "JetBrains Mono", "Cascadia Code", monospace
```

规则：

- 一级标题可以更有书卷气，正文保持高可读性。
- 不用过细字重；中文正文避免 300 以下字重。
- 不用负字距。
- 移动端不使用过大的标题。

## 页面结构

主站是知识库，不做传统营销首页。

首页首屏应该直接回答：

- 这是什么
- 给谁看
- 从哪里开始读
- 如何贡献

不要做大面积口号页，也不要把正文藏在装饰卡片后面。

## 组件原则

### 导航

- 左侧导航服务长期查找，不镜像所有文件。
- 一级目录清楚，二级目录适度展开。
- C 类个人文章不逐篇进入主导航。
- `navigation.indexes` 下，任何分组如需点击到自己的总览页，必须把该分组的 `index.md` 显式放在导航组第一项，不能让主题猜默认页。
- 当前阶段可以把稳定骨架目录放入导航，方便贡献者找到位置；等个人文章增多后，C 类文章仍由索引页承载，不逐篇塞进主导航。
- 左侧导航不使用白色背景块、发光阴影、侧边竖条或边框高亮表示当前状态。
- 当前状态只用文字颜色、字重和适度行距表达；导航不能挤成一团。
- 导航层级必须反映内容架构，不为了少点几下而把不同层级对象拍平成并列项。

### 内容页

- 正文宽度控制在适合中文阅读的范围。
- 长文用小标题、提示块、表格、案例引用分段。
- 案例引用要保留背景和限制条件，不能只摘成功结论。

### 索引页

索引页不是卡片墙，优先使用分组列表：

```text
标题
一句话说明
适合谁
已有样本
缺什么
如何贡献
```

卡片只用于少量重点入口，不要全站卡片化。

首页和一级目录优先使用 `topic-list` 式布局：

- 左侧编号提供定位。
- 中间是目录对象。
- 右侧是状态或贡献提示。
- 整行可点击，移动端折叠为两行，但不隐藏状态。

不要为了区分类别而加侧边色条。需要区分时，优先使用编号、文字层级、行距和轻量状态文本。

### 提示块

提示块只用于真正需要强调的信息：

- 公开边界
- 适用年份
- 风险提醒
- 投稿说明
- 缺口征集

不要每段都用 admonition。

公开转载、评论边界、建设中状态不默认使用 quote 式提示框。优先使用：

- 首页脚注式 `fineprint`
- 目录页顶部的细线状态条
- 空页面的轻量 empty state
- 模板页中的短说明

如果一个提示块看起来像“AI 把一句话装进了引用框”，就应该降级成正文、脚注或元信息。

### 图片

- 正文图片要服务解释，不做气氛装饰。
- 图片放在拥有它的页面或目录旁边的 `_assets/`，不要统一堆进全站 `assets/`。
- 图片说明优先使用 `figure` / caption；截图、流程图、地图要说明来源、时间和适用边界。
- 长图、截图和流程图在移动端必须能自然缩放，不让文字小到不可读。
- 除全站 logo、社交封面和主题资源外，不使用 `docs/assets/` 存放正文图片。

## 公开转载物料

公开转载物料可以比主站更有叙事感，但不能改变主站归属，也不能改变作者署名、匿名和脱敏约定。

转载物料的视觉应沿用：

- 暖纸色背景
- 深墨蓝标题
- 少量手册式分隔线
- 不用网红感渐变封面
- 不用夸张情绪标题

公开转载可以做摘要和重新排版，但主站仍是长期正本。

## 禁止事项

- 不做纯宣传落地页
- 不使用大面积渐变背景
- 不用玻璃拟态、发光边框、悬浮大卡片堆叠
- 不用左侧粗色条、侧边强调线、blockquote 式边框作为默认视觉手段
- 不让图标替代内容层次
- 不把每个页面做成公众号文章
- 不把敏感内容用设计包装后公开

## 第一版 CSS 落点

后续落地时，优先在 `docs/stylesheets/extra.css` 中实现：

- 暖纸色背景
- 中文正文宽度与行距
- 标题字体和层级
- 链接颜色
- 表格和案例引用样式
- 投稿提示块样式

第一版不要改 MkDocs Material 的核心模板，先用主题配置和少量 CSS 完成气质统一。
